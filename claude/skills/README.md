# 스킬 카탈로그

규칙(`../rules/`)이 "우리 팀은 이렇게 한다"는 문서라면, 스킬은 그 규칙을 Claude가 대신 실행해
주는 자동화다. 개발자가 규칙을 다 외우지 않아도, 하려는 작업만 말하면 스킬이 규칙에 맞는
결과물을 만들어 준다. 무엇부터 할지 모르겠으면 `dev-workflow`부터 부르면 된다.

각 스킬은 `<이름>/SKILL.md`에 들어 있고, `templates/`가 있는 스킬은 복사해서 바로 쓸 수 있는
실물 파일을 품고 있다.

---

## dev-workflow — 시작점(오케스트레이터)

"뭐부터 해야 하지?" 싶을 때 여기서 시작하면 알맞은 규칙과 스킬로 안내해 주는 안내데스크다.

- **언제**: 작업 유형이 애매하거나, 합류한 지 얼마 안 돼 흐름을 모를 때.
- **무엇을**: 먼저 공통 점검(범위 확인·가드레일·브랜치)을 하고, 작업 유형을 판별해 맞는 스킬로
  넘긴 뒤, 작업이 끝나면 검증 → 커밋 → PR → CI까지 표준 경로로 안내한다. 문서 안에 전체
  흐름도가 들어 있다.
- **결과물**: 코드를 직접 만들지는 않는다. "이 작업은 create-slice를 쓰고 rules 20·40을 보면
  된다"처럼 길을 잡아 준다.

## plan-feature — 큰·모호한 작업 계획

코드부터 짜지 않고, 무엇을 만들지부터 또렷하게 맞춘다.

- **언제**: 요청이 모호하거나(특히 비개발자 요청), 3파일 이상·구조 변경·되돌리기 어려운 작업일 때.
- **무엇을**: 일상어로 인터뷰(누가·무엇을·왜·예외·완료 기준) → 우리 구조(슬라이스·컴포넌트·API·
  테스트)로 번역한 **단계 계획** → 스스로 빈틈 점검 → 일상어 미리보기로 **확인**받은 뒤, 각 단계를
  build 스킬(`create-slice`·`create-component`·`gen-api-types`·`write-test`)에 넘긴다.
- **결과물**: 코드가 아니라 합의된 계획. 큰 작업이 옆길로 새거나 헛돌지 않게 한다.

## setup-fe-project — 새 프로젝트 세팅

빈 저장소를 팀 표준 모노레포로 처음부터 세운다.

- **언제**: 새 프로젝트를 시작할 때.
- **무엇을**: pnpm workspace + Turborepo 모노레포 골격(`apps/admin`, `apps/service`,
  `packages/ui·config`)을 만들고, 품질 게이트(oxlint·prettier·husky·commitlint·size-limit)와
  각 앱의 FSD 구조·eslint·테스트 툴링을 배치한다. 설정 파일은 손으로 쓰지 않고 스킬에 포함된
  `templates/`의 실물 파일을 복사한다 — 그래서 누가 세팅해도 결과가 같다.
- **결과물**: `pnpm install → lint → type-check → test → build`가 통과하는, 바로 개발 가능한 프로젝트.

## create-component — UI 컴포넌트 추가

컴포넌트를 팀 표준 패턴으로, 스토리와 테스트까지 한 세트로 만든다.

- **언제**: 버튼·카드·모달 같은 UI를 추가할 때.
- **무엇을**: `interface Props` + `withDefaults`로 props를 정의하고, 변형은 `cva`, 외부에서 넘어온
  class는 `cn`으로 병합한다(Tailwind만 사용, 다크모드는 `dark:`). `templates/BaseButton`을 복사
  시작점으로 쓴다.
- **결과물**: `<Name>.vue` + 스펙(렌더·prop·이벤트·클래스 병합 검증) + 스토리 + `index.ts` export.
  누가 만들어도 같은 구조와 같은 테스트 수준의 컴포넌트가 나온다.

## implement-from-figma — Figma 디자인 구현

Figma 디자인을 상상 없이 정확히 코드로 옮긴다.

- **언제**: Figma 링크/노드를 받아 화면·컴포넌트를 만들 때. "이 디자인대로 만들어".
- **무엇을**: Figma MCP로 디자인 컨텍스트·스크린샷·변수를 읽고 → 기존 컴포넌트(Code Connect/
  `shared/ui`)를 먼저 재사용 → Figma 값을 우리 디자인 토큰에 매핑(하드코딩 금지) → `create-component`/
  `create-slice`로 우리 컨벤션(cva/cn·a11y·i18n)대로 구현 → 스크린샷과 시각 대조.
- **결과물**: 디자인과 일치하면서 우리 규칙을 지키는 컴포넌트/화면. MCP 자동 코드를 그대로 붙이지 않는다.

## create-slice — 화면·기능·엔티티 추가

Feature-Sliced Design 규칙에 맞는 폴더 한 덩어리(슬라이스)를 만든다.

- **언제**: 새 페이지, 기능, 비즈니스 엔티티를 추가할 때.
- **무엇을**: 계층을 고르고(entities=명사, features=액션, widgets=조합 UI, pages=라우트),
  필요한 세그먼트만 만든다 — `ui/`, `model/`(클라이언트 스토어: Pinia/Zustand), `api/`(서버 쿼리 훅:
  Vue Query/React Query). 바깥으로는
  `index.ts`로 public API만 노출한다. page면 라우터에 지연 로딩으로 등록한다.
- **결과물**: 올바른 위치에 놓인 슬라이스. 계층 import 방향은 eslint가 강제하므로, 구조를 몰라도
  규칙 위반은 빌드에서 걸린다.

## gen-api-types — 서버 API 연동

백엔드 OpenAPI 명세에서 타입을 자동 생성하고 프레임워크의 Query 훅(Vue Query/React Query)을 붙인다.

- **언제**: 백엔드 API가 바뀌거나 새 엔드포인트를 연동할 때.
- **무엇을**: 명세를 가리켜 `pnpm run gen:api`로 `schema.d.ts`를 재생성하고(이 파일은 손으로
  고치지 않는다), 타입세이프 클라이언트로 `useQuery`/`useMutation` 훅을 작성한다(쿼리 키도 함께 export).
- **결과물**: 백엔드와 어긋날 일이 없는 타입 안전한 API 훅. 요청·응답 타입을 직접 적지 않는다.

## gen-tokens — 디자인 토큰 생성

디자이너의 Token Studio export를 Tailwind 디자인 토큰으로 동기화한다.

- **언제**: 디자이너가 새 토큰(색상·간격·radius·타이포)을 넘겼거나 토큰을 바꿔야 할 때.
- **무엇을**: `tokens/`에 export 원본을 덮어쓰고 `pnpm run gen:tokens`로 `src/app/styles/tokens.css`를
  재생성한다(생성 파일 — 손으로 수정 금지). `$type` 기준으로 Tailwind 네임스페이스에 매핑되고
  light/dark가 갈린다. 배경은 `docs/adr/0010`.
- **결과물**: 디자이너 토큰과 일치하는 Tailwind 유틸. (현재 파이프라인은 청사진 상태 — SKILL의 재설치 절차 참고.)

## write-test — 테스트 작성

기존 코드에 테스트를 알맞은 층으로 추가한다.

- **언제**: 함수·스토어·컴포넌트에 테스트를 붙이거나, 통합·E2E를 별도로 작성할 때.
- **무엇을**: 대상에 따라 층을 고른다 — 유틸/스토어는 단위, 컴포넌트는 렌더 도구(Vue=`@vue/test-utils`,
  React=Testing Library),
  API 흐름은 MSW 통합, 사용자 플로우는 Playwright E2E. 구현이 아니라 동작을 검증하고, 파괴적
  가드 변경엔 회귀 스펙과 반대 방향 케이스를 넣는다.
- **결과물**: 소스 옆(또는 `e2e/`)에 놓인, 리팩터링에 잘 안 깨지는 테스트.

## debug-issue — 버그 디버깅

추측으로 고치지 않고, 재현하고 원인을 확인한 뒤 최소한으로 고친다.

- **언제**: "안 돼요", 에러, 이상 동작을 조사하고 고칠 때.
- **무엇을**: 재현(입력·환경·기대 vs 실제) → 코드 경로로 가설 → 로그/테스트로 **먼저 검증** →
  확인된 원인만 최소 수정(범위 밖 리팩터 금지) → 재발 막는 회귀 테스트 추가. 가설이 틀리면
  땜질 대신 되돌리고 다시 본다.
- **결과물**: 원인이 확인된 수정 + 회귀 테스트.

## create-adr — 결정 기록

중요한 아키텍처·도구 결정을 문서로 남긴다.

- **언제**: 스택·구조·도구를 바꾸는 결정을 할 때.
- **무엇을**: `docs/adr/`에 다음 번호로 템플릿을 복사해 배경·결정·결과·검토한 대안을 적고, 목록
  표를 갱신한다. 이전 결정을 대체하면 옛 문서를 `superseded`로 표시하되 본문은 고치지 않는다.
- **결과물**: "왜 이렇게 정했는지"가 남는 기록. 나중에 합류한 사람이 맥락을 이해할 수 있다.

## review-changes — 변경 리뷰

현재 변경을 팀 규칙에 비춰 리뷰하고 지적을 심각도별로 정리한다.

- **언제**: 커밋/PR 전 자기 리뷰, 또는 남의 변경을 검토할 때.
- **무엇을**: diff를 파악하고 범위·아키텍처(FSD 경계)·컴포넌트·상태/API·테스트·에러·보안·접근성
  차원으로 점검해 `file:line` + 심각도로 지적한다. 확신 없는 지적은 단정하지 않는다.
- **결과물**: PR 전에 잡는 문제 목록. High가 있으면 먼저 해결하도록 권고한다.

## prepare-commit — 커밋 정리

커밋 직전에 안전 점검을 하고 규약에 맞는 메시지를 만든다.

- **언제**: 커밋할 때.
- **무엇을**: 먼저 브랜치를 확인해 `main`·`develop` 같은 보호 브랜치면 직접 커밋을 막고 feature
  브랜치로 안내한다. 이어서 범위·관심사·중복·재사용을 점검하고 lint·type-check·test를 확인한 뒤
  Conventional Commits 메시지를 만든다. 실제 커밋 시에는 husky 훅(pre-commit의 lint-staged,
  commit-msg의 commitlint)이 한 번 더 강제한다.
- **결과물**: 규약을 지킨 안전한 커밋. 실수로 보호 브랜치에 커밋하는 사고를 막는다.

## prepare-pr — PR 준비

PR 올리기 직전 검증과 설명 작성을 대신한다.

- **언제**: 커밋을 마치고 PR을 올릴 때.
- **무엇을**: 검증 runbook(test→type-check→lint→build→size→e2e)을 돌리고, PR 설명을
  "무엇을/왜/어떻게 검증" 템플릿으로 채운 뒤 리뷰어가 볼 지점을 요약한다. 보호 브랜치로 직접
  머지하지 않고 푸시·PR 생성 명령을 안내한다.
- **결과물**: 검증이 끝난, 리뷰하기 좋은 PR 초안.
