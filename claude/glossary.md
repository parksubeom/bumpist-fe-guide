# 용어 사전

이 가이드에 나오는 말들을 쉬운 말로 풀어 둔다. 개발이 처음이어도 이 사전만 곁에 두면 나머지
문서(규칙·스킬)를 그대로 읽을 수 있다. "쉽게 말하면"만 읽어도 충분하다.

## 화면을 만드는 말

- **컴포넌트(component)** — 화면을 이루는 조각 하나. 버튼, 입력창, 카드처럼 재사용하는 부품.
  쉽게 말하면 "화면 조각".
- **슬라이스(slice)** — 하나의 기능/화면을 이루는 파일 묶음(화면 조각 + 데이터 + 서버 연결).
  쉽게 말하면 "기능 한 덩어리". 도메인 명사(cart·product) 하나가 슬라이스 하나.
- **세그먼트(segment)** — 슬라이스 안을 역할별로 나눈 칸: `ui`(화면)·`model`(상태)·`api`(서버 연결)·
  `lib`(순수 계산). 쉽게 말하면 "기능 덩어리 안의 역할 칸".
- **lib(순수 계산)** — 입력이 같으면 결과도 늘 같고 바깥(화면·저장소)을 건드리지 않는 함수 모음.
  테스트가 쉬워 따로 모아 둔다. 쉽게 말하면 "계산기 함수".
- **레이어/계층(layer)** — FSD의 위아래 층(app → pages → widgets → features → entities → shared).
  위층만 아래층을 가져다 쓴다. 쉽게 말하면 "폴더의 위계".
- **페이지(page)** — 주소(URL) 하나에 대응하는 화면 전체. 쉽게 말하면 "한 화면".
- **위젯(widget)** — 여러 기능을 조합한 큰 화면 블록(예: 사이드바, 헤더).
- **FSD (Feature-Sliced Design)** — 파일을 "역할별 층"으로 정리하는 규칙. 위층은 아래층만
  가져다 쓸 수 있어 구조가 안 엉킨다. 쉽게 말하면 "폴더 정리 규칙".
- **prop(프롭)** — 컴포넌트에 건네는 설정값(예: 버튼 색, 크기). 쉽게 말하면 "부품 옵션".

## 데이터·상태를 다루는 말

- **상태(state)** — 지금 화면이 기억하고 있는 값(예: 입력한 검색어, 열림/닫힘).
- **Pinia(피니아)** — 화면들이 공유하는 상태를 담아 두는 곳. 쉽게 말하면 "앱 안 메모".
- **서버 상태 / Vue Query** — 서버에서 받아 온 데이터(목록, 사용자 정보)를 관리하는 도구.
  "다시 불러오기·캐시"를 알아서 해 준다.
- **API** — 프론트엔드(화면)와 백엔드(서버)가 주고받는 약속된 창구.
- **타입(type) / TypeScript** — 값의 종류(숫자·글자 등)를 미리 정해 실수를 막는 장치.
  쉽게 말하면 "오타·실수 예방기".

## 모양(스타일) 관련 말

- **Tailwind(테일윈드)** — 화면 꾸밈을 `class` 이름으로 하는 방식(예: `p-4`=안쪽 여백).
- **다크모드** — 어두운 테마. `dark:`로 표시.
- **디자인 토큰** — 색·간격 같은 값을 이름으로 정해 둔 것(예: `bg-bg-default`). 디자이너와
  숫자로 실랑이하지 않게 해 준다.
- **i18n(국제화)** — 화면 문구를 언어별로 관리해 다국어를 지원하는 것.
- **Figma(피그마)** — 디자이너가 화면을 그리는 도구. 우리 화면의 "정답 그림".
- **Figma MCP** — Claude가 Figma 디자인을 직접 읽어 오는 연결 통로(색·간격·컴포넌트 정보).
  쉽게 말하면 "디자인을 코드로 가져오는 창구".
- **Code Connect** — Figma의 디자인 컴포넌트와 우리 코드 컴포넌트를 이어 둔 매핑. 있으면 그 코드를
  그대로 재사용.

## 품질·테스트 관련 말

- **테스트(test) / 스펙(spec)** — 코드가 의도대로 도는지 자동으로 확인하는 코드.
  쉽게 말하면 "자동 점검".
- **Vitest / Playwright / Storybook** — 각각 부품 점검 / 실제 브라우저 점검 / 부품 미리보기 도구.
- **린트(lint) / ESLint / oxlint** — 코드 규칙 위반을 자동으로 잡아 주는 검사기.
- **포맷(format) / Prettier** — 코드 모양(줄바꿈·따옴표)을 자동으로 통일해 주는 도구.
- **번들(bundle) / size-limit** — 사용자에게 보내는 코드 묶음. 너무 커지지 않게 예산으로 관리.

## 협업·git 관련 말

- **git** — 코드 변경 이력을 저장·되돌리는 도구. 쉽게 말하면 "무한 저장·되돌리기".
- **커밋(commit)** — 변경을 한 묶음으로 저장하는 것. 쉽게 말하면 "저장 지점 찍기".
- **브랜치(branch)** — 본 줄기를 건드리지 않고 따로 작업하는 갈래. 실험해도 안전.
- **main / 보호 브랜치** — 최종본이 모이는 본 줄기. 함부로 직접 넣지 않고 검토를 거친다.
- **PR (Pull Request)** — "이 변경을 본 줄기에 넣어도 될까요?" 하고 검토를 요청하는 것.
  쉽게 말하면 "검토 요청".
- **리뷰(review)** — 다른 사람(또는 AI)이 변경을 점검하는 것.
- **CI** — PR을 올리면 자동으로 테스트·검사를 돌려 통과해야 넣어 주는 관문. 쉽게 말하면
  "자동 검문소".
- **훅(hook) / husky** — 커밋·푸시 같은 순간에 자동으로 검사를 끼워 넣는 장치.
- **머지(merge)** — 갈래(브랜치)의 변경을 본 줄기에 합치는 것.
- **submodule(서브모듈)** — 다른 저장소를 내 프로젝트 안에서 참조로 끌어다 쓰는 것.

## 도구·구조 관련 말

- **Vue / React** — 화면을 만드는 두 프레임워크. 프로젝트마다 **하나**를 고른다(둘 다 지원). 공통
  규칙은 같고, 프레임워크 특화 규칙만 `rules/vue/`·`rules/react/`로 나뉜다.
- **Pinia / Zustand** — 클라이언트 상태 저장소. Vue는 Pinia, React는 Zustand(역할 동일).
- **Vue Query / React Query** — 서버 데이터 관리 도구(둘 다 TanStack Query 계열).
- **모노레포(monorepo)** — 여러 앱(관리자·서비스 등)을 한 저장소에 묶어 관리하는 방식.
- **pnpm / Turborepo** — 모노레포의 패키지 설치·빌드를 빠르게 해 주는 도구.
- **CLAUDE.md** — Claude(AI)가 작업 전에 읽는 프로젝트 지침 파일.
- **스킬(skill) / 규칙(rule)** — 규칙은 "우리 팀은 이렇게 한다"는 문서, 스킬은 그 규칙을 AI가
  대신 실행해 주는 자동 절차. (이 가이드의 핵심 두 축)
