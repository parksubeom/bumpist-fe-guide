# 기여 가이드

관심 가져줘서 고맙다. 이 문서는 일부러 짧다 — **원칙 하나, 흐름 하나, 규약 하나**만 지키면 된다.

## 허브 원칙 — 무엇을 받고, 무엇을 안 받나

이 저장소는 여러 프로젝트에 두루 쓰는 **공통 표준의 단일 출처(허브)**다.

**받는 것**

- 공통 규칙(`rules/*.md`)과 프레임워크 트랙(`rules/vue|react|next/`) 개선
- 스킬(`skills/*`) 절차 개선, 앱 템플릿(`skills/setup-fe-project/templates/app*`) 버그 수정
- 실전에서 밟은 함정(지뢰) 반영 — **재현 방법과 함께** (v0.5.1의 Next 트랙 픽스 3종이 좋은 예다)
- 문서·오타·링크 수정

**받지 않는 것**

- **특정 프로젝트/도메인 전용 프롬프트·스킬** — 그건 그 프로젝트의 `.claude/`에 둔다.
  여러 프로젝트에서 두루 쓸 만해지면, 그때 이슈로 허브 승격을 제안한다.
- 근거 없는 스택·구조 교체 제안 — 필요성·대안 비교·영향을 밝혀야 한다(`rules/10-guardrails.md` 참조).

## 흐름 — 이슈 먼저, PR은 그다음

1. **이슈부터 연다.** 버그면 「버그 리포트」, 새 규칙·스킬 아이디어면 「규칙/스킬 제안」 템플릿을 쓴다.
   오타 수준의 작은 수정은 이슈 없이 바로 PR을 열어도 된다.
2. 방향이 맞다고 확인되면 **포크 → 브랜치**에서 작업한다. 브랜치 이름은 `feat/…`, `fix/…`, `docs/…` 식.
3. **PR을 연다.** PR 템플릿의 체크리스트를 채운다. 특히 — 규칙을 바꾸면 그 규칙을 참조하는
   스킬·README 표도 함께 맞춘다(문서·CLI·스킬·규칙의 정합이 이 허브의 생명이다).
4. 리뷰 후 머지한다. **배포(git 태그 + npm publish)는 메인테이너가 한다.**
   버전은 semver를 따르고, 기존 규칙·스킬이 실제로 바뀔 때만 올린다 —
   자세한 기준은 [README의 버전 관리](./README.md#버전-관리-semver)와 [CHANGELOG.md](./CHANGELOG.md).

## 커밋 규약 — 한글 + Conventional Commits

- 형식: `type: 한글 요약` — 예: `fix: Next 트랙 vitest 수집 범위를 src로 한정`
- type: `feat` | `fix` | `docs` | `refactor` | `chore` | `test`
- 본문(선택)도 한글로, 무엇을·왜 바꿨는지 적는다.
- 커밋에 `Co-Authored-By` 등 트레일러는 붙이지 않는다.

## 보내기 전 확인

문서 중심 저장소라 빌드 게이트는 없지만, 최소한 이것은 확인한다.

- 템플릿 파일(`.ts`/`.tsx`/`.json`)을 고쳤다면 구문이 유효한가
- 문서 안 상대 링크·파일 참조가 깨지지 않는가
- 규칙 ↔ 스킬 ↔ README 표가 서로 어긋나지 않는가

막히면 이슈로 물어보면 된다. 작은 기여도 환영한다.
